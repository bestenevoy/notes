# 大模型工具调用问题

## 一、工具调用不稳定（超时、报错、漏调用、重复调用、参数错误）
### 核心根因
1. LLM输出工具格式错乱（JSON残缺、括号缺失、字段写错）；
2. 未做调用前置校验、超时重试机制缺失；
3. 上下文过长，模型遗忘工具入参规范；
4. 并发/网络波动、第三方接口限流。

### 解决方案
#### 1. 格式层强制约束（提示词+解析器）
- 固定强格式模板：强制仅输出标准JSON工具调用块，禁止自然语言穿插参数；
- 增加格式校验拦截器：后端写JSON解析逻辑，解析失败自动重试重生成，丢弃错乱输出；
- 工具白名单：限定可调用工具名称，模型输出未知工具直接驳回。

#### 2. 调用流程工程加固
- 超时、失败自动重试：限2次重试，防止无限循环；
- 幂等调用：给每轮工具调用生成唯一traceId，拦截重复请求；
- 参数校验器：枚举每个工具必填/可选参数、数据类型、取值范围，参数缺失/类型错误直接返回修正提示给模型重新生成。

#### 3. 上下文优化
- 工具调用历史做精简压缩，只保留工具入参+返回结果，删除冗余对话；
- 系统提示词置顶工具规范，每次调用前强制读取工具定义。

## 二、工具幻觉（无依据编造工具、编造参数、编造工具返回结果、没调用工具却虚构数据）

### 核心根因
模型预训练知识强于工具使用逻辑，缺少“必须调用工具才能获取外部信息”的约束，存在自主脑补倾向。

### 分层解决
1. **提示词硬约束（基础）**
    强制规则写入System Prompt：
    - 所有实时/外部/事实数据**必须调用对应工具**，禁止自行编造数字、时间、文档内容、接口返回；
    - 未执行工具前，严禁输出任何外部查询结果；
    - 明确惩罚逻辑：若编造数据视为违规，需要重新执行工具查询。
2. **运行时拦截（核心）**
    - 幻觉识别规则：检测回答中出现外部数据（价格、日期、数据库内容、网页信息）但无对应工具调用记录，直接阻断输出，强制重新走工具流程；
    - 工具结果隔离：模型只能基于工具返回文本作答，禁止混合自身知识库内容覆盖工具结果。
3. **后校验模块**
    输出后置校验：抽取回答中事实类信息，反向匹配工具返回内容，不匹配则驳回重生成。

## 三、危险操作（删库、文件删除、系统命令、高危接口、数据删除/篡改、隐私批量导出）
### 根因
工具未分级、缺少权限管控、无高危指令识别、未做人工拦截。

### 四层防护体系
1. **工具分级管控**
    - 分为只读工具（查询、检索、读取文件）、高危写操作（删除、修改、执行命令、批量导出）；
    - 普通会话默认禁用高危工具，需管理员二次授权才可开启。
2. **高危指令识别拦截**
    关键词+语义识别：识别删除、清空、销毁、rm、drop、delete、批量导出用户数据、执行shell等意图，直接拒绝调用。
3. **操作白名单+范围限制**
    - 文件操作限定指定目录，禁止访问系统根目录、隐私目录；
    - 数据库操作仅允许SELECT查询，拦截DROP/DELETE/ALTER等修改语句；
    - 批量操作加数量上限，单次最多处理少量数据。
4. **审计留痕**
    所有高危工具调用强制记录操作者、时间、入参、操作结果，支持事后溯源。

## 四、越权操作（超出用户需求调用工具、跨权限访问数据、主动调用无关工具、多工具无意义连环调用）
### 根因
模型缺少“按需调用”约束，容易过度推理、主动拓展无关操作。

### 解决方案
1. 提示词约束：**仅调用满足用户当前问题必需的最小工具集**，禁止额外查询无关信息；用户未要求不主动调用工具。
2. 调用次数限制：单轮问答最多N次工具调用（如3次），防止无限连环调用越权查询；
3. 意图匹配校验：解析用户原始需求，工具调用目的和用户意图不匹配则驳回；
    例：用户只查商品价格，模型调用用户隐私订单接口，直接拦截。
4. 数据域隔离：用户A的工具会话不能访问用户B的数据，增加租户/用户ID鉴权。

## 五、不遵循指令（无视工具规则、无视用户限制、忽略前置约束、拒绝调用必要工具）
### 根因
系统提示词权重低、上下文冲淡约束、模型偏好偷懒不调用工具。

### 完整方案
1. **提升系统指令权重**
    - 把工具规则、用户限制放在对话最开头，且每次工具生成前重复精简版约束；
    - 使用强制语气规则，明确违反指令会重新生成。
2. **意图强制分流**
    前置分类器判断问题类型：事实查询/计算/文件读取类，强制走工具通道，禁止直接回答。
3. **违规重生成机制**
    检测到模型无视规则（能查工具却直接回答、无视用户禁止条款），自动回滚，重新生成并追加约束提示。
4. **少样本示例注入**
    在系统prompt加入正反示例：
    - 正确示例：用户问实时数据→调用工具；
    - 错误示例：直接编造数据→违规，需要重新调用工具。

# 通用兜底整体架构（落地可用）
整套链路加多层拦截，从输入到输出闭环解决全部问题：
1. **输入层**：意图识别、高危指令拦截、越权意图预判；
2. **模型层**：强约束系统提示词+工具定义+正反示例；
3. **工具调用中间件（核心）**
    - 格式解析器、参数校验器、重试机制、幂等控制；
    - 工具权限分级、高危操作鉴权；
4. **结果层**：幻觉校验、指令合规校验、越权检测；
5. **输出层**：审计日志、违规告警、失败自动重试/驳回。

# 轻量化低成本方案（无复杂工程时，仅靠提示词优化）
如果暂时无法开发后端中间件，仅通过Prompt即可大幅缓解：
1. 完整罗列全部工具、入参、调用场景；
2. 逐条写明禁止行为：编造数据、过度调用、高危操作、忽略指令；
3. 限定单次最大调用次数；
4. 要求严格JSON输出，附带解析失败处理规则；
5. 结尾强制重申：所有外部信息必须调用工具，禁止脑补。
